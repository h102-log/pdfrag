# 이 프로젝트, 쉽게 이해하기

> 대상: 코드를 잘 모르는 분 / 웹개발은 알지만 AI·RAG는 처음인 분
> 목적: "지금까지 만든 게 도대체 뭐고, 어떻게 돌아가는가"를 그림처럼 이해하기

---

## 1. 한 문장 요약

**"수천 페이지짜리 설비 매뉴얼·작업절차서(PDF, 워드, 표 잔뜩)를 AI에게 먹여두고, 사람이 자연어로 질문하면 정확한 답 + 그 답의 출처(어느 문서 몇 페이지)까지 짚어주는 시스템"** 을 만드는 중입니다.

예를 들어 현장 엔지니어가 이렇게 묻습니다:

> "베어링 진동이 한계를 넘으면 어떤 설비를 멈춰야 해?"

그러면 시스템이:

> "**모터 M-22**를 정지시켜야 합니다. (출처: 펌프 매뉴얼 〈4. 이상 시 처리 절차〉)"

이렇게 답하고, 옆에 원본 문서의 해당 부분을 하이라이트해서 보여주는 게 최종 목표입니다.

---

## 2. 왜 그냥 ChatGPT한테 물어보면 안 되나? (RAG 개념)

ChatGPT 같은 AI는 똑똑하지만 **"우리 회사 설비 매뉴얼"은 모릅니다.** 학습한 적이 없으니까요. 그리고 모르는 걸 물으면 그럴듯하게 지어냅니다(이걸 **환각/할루시네이션**이라고 부릅니다). 설비 정비 현장에서 AI가 답을 지어내면 사고로 이어집니다.

그래서 쓰는 기법이 **RAG (Retrieval-Augmented Generation)** 입니다. 풀어 쓰면:

1. **Retrieval(검색)**: 질문이 들어오면, 먼저 우리 문서 더미에서 **관련 있는 부분만 찾아옵니다.**
2. **Augmented(보강)**: 찾아온 그 내용을 AI에게 같이 건넵니다 — "이 자료를 근거로 답해" 라고.
3. **Generation(생성)**: AI는 **준 자료 안에서만** 답을 만듭니다.

> 웹개발 비유: AI한테 그냥 묻는 건 "구글 없이 머릿속 지식으로만 답하기"이고,
> RAG는 "먼저 사내 위키를 검색해서 그 문서를 보면서 답하기"입니다.
> 검색해서 근거를 깔아주니 거짓말(환각)이 확 줄어듭니다.

### 이 프로젝트가 한 발 더 나간 부분 — GraphRAG

보통 RAG는 문서를 잘게 잘라 **"비슷한 문장 찾기(벡터 검색)"** 로만 자료를 찾습니다. 그런데 이건 약점이 있습니다. 예를 들어:

> "베어링 진동이 한계를 넘으면 어떤 설비를 멈춰?"

이 질문의 답은 한 문장에 안 들어 있습니다. **여러 정보를 연결**해야 나옵니다:

- 베어링(B-7)은 → 펌프(P-101)를 **지지**한다
- 펌프(P-101)는 → 모터(M-22)가 **구동**한다
- 진동 한계 초과 시 → 절차상 **모터를 정지**한다

이렇게 "A는 B와 이런 관계, B는 C와 저런 관계" 식으로 **점과 선으로 이어진 지도**를 만들어 두면, 여러 단계를 건너뛰는 질문(멀티홉 질문)에도 답할 수 있습니다. 이 지도를 **지식 그래프(Knowledge Graph)** 라 하고, 그걸 활용하는 RAG를 **GraphRAG**라고 부릅니다.

> 비유: 일반 RAG = 책에서 비슷한 문장 찾기. GraphRAG = 등장인물 관계도를 그려놓고 추리하기.

## 3. 문서 한 장이 시스템을 통과하는 여정 (핵심 파이프라인)

지금까지 만든 핵심은 **"문서를 넣으면 → 지식 그래프가 만들어진다"** 는 컨베이어 벨트입니다. 단계별로 봅니다.

```
[원본 파일]          [① 로딩]          [② 청킹]            [③ 추출]              [④ 저장]
pump_manual.md  →  글/표로 분해  →  적당한 크기로 자름  →  AI가 관계 뽑음  →  그래프 DB에 적재
pump_spec.pdf
pump_report.docx
```

### ① 로딩 (Loading) — 파일을 읽어 표준 형태로 바꾸기

PDF, 워드(DOCX), 마크다운(MD) 등 **파일 형식이 제각각**입니다. 형식마다 읽는 방법이 다릅니다. 그래서 "어떤 형식이든 읽어서 **똑같은 모양**으로 바꿔주는" 부품을 만들었습니다.

- 결과물은 전부 `LoadedDoc`(문서 한 개)이라는 같은 형태가 됩니다. 그 안에는 `Element`(글 덩어리 또는 표 하나)들이 들어 있습니다.
- **중요**: 읽을 때 "이 내용이 원본의 **어디**에서 왔는지"(PDF는 몇 페이지, MD는 어느 섹션)를 같이 기록합니다. 나중에 출처를 짚어주려면 이 위치 정보를 끝까지 들고 가야 하기 때문입니다.

> 웹개발 비유: 여러 외부 API의 제각각인 응답(JSON 구조가 다 다름)을 우리 내부 공용 DTO 하나로 정규화하는 어댑터 계층과 같습니다.

### ② 청킹 (Chunking) — 적당한 크기로 자르기

문서 전체를 통째로 AI에 주면 너무 크고 비쌉니다. 그래서 **의미 단위로 적당히 자릅니다(=청크).** 단순히 글자 수로 자르면 문장이 중간에 끊기니 규칙을 둡니다:

- **글**은 문장 경계에서 약 800자 단위로 자르되, 앞뒤 청크가 120자씩 **겹치게** 합니다(맥락 유실 방지).
- **표는 절대 쪼개지 않습니다.** 표를 중간에서 자르면 "이 숫자가 무슨 항목인지" 알 수 없게 되니까요. 표가 너무 크면 행 단위로 나누되 **머리글 행을 매 조각에 반복**해 넣어, 어느 조각이든 혼자서도 읽을 수 있게 합니다.
- 잘린 모든 조각은 자기 출처 정보(문서명/위치/다운로드 링크 등)를 그대로 들고 다닙니다.

> 왜 중요? 이게 답변 품질의 8할입니다. 표가 깨지거나 문맥이 끊기면 AI가 헛소리를 합니다.

### ③ 추출 (Extraction) — AI가 "관계"를 뽑아내기

자른 글 조각을 **Claude(AI)** 에게 주고 이렇게 시킵니다:

> "이 글에서 설비/부품 같은 **대상(엔티티)** 과 그들 사이의 **관계** 를 뽑아줘."

그러면 AI가 이런 식의 **삼각 정보(트리플릿)** 를 만들어 냅니다:

```
(펌프 P-101) ―[구동]→ (모터 M-22)
(펌프 P-101) ―[지지]→ (베어링 B-7)
(베어링 B-7) ―[점검]→ (진동)
```

- 우리는 AI에게 도메인 힌트(설비/펌프/모터/베어링… 같은 단어, 구동/지지/점검… 같은 관계)를 줍니다. 단, **강제는 아니고 유도만** 합니다("열린 스키마"). 너무 빡빡하게 막으면 관계를 적게 뽑는 함정이 있어서, 느슨하게 힌트만 줍니다.

> 웹개발 비유: 비정형 텍스트를 받아 구조화된 레코드(주어-관계-목적어)로 바꾸는 파서인데, 그 파서가 정규식이 아니라 AI입니다.

### ④ 저장 (Storing) — 그래프 DB에 적재

뽑아낸 점(엔티티)과 선(관계)을 **Neo4j**라는 **그래프 전용 데이터베이스**에 저장합니다.

- 동시에, ②에서 만든 글 조각 자체도 "출처 노드"로 저장하고, AI가 뽑은 엔티티가 **어느 글 조각에서 나왔는지** 연결(`MENTIONS`)해 둡니다. → 그래서 나중에 "이 답의 근거가 어느 문장이었나"를 역추적할 수 있습니다.
- 글 조각의 ID를 그래프 노드의 ID로 그대로 씁니다. 이건 **나중에 만들 벡터 검색(다음 단계)과 같은 열쇠로 연결**하기 위한 포석입니다.

> 웹개발 비유: SQL이 표(행/열)에 강하다면, 그래프 DB는 **"관계 타고 이동하기"** 에 강합니다.
> `MATCH (펌프)-[:구동]->(모터)` 처럼 관계를 따라 질의합니다(쿼리 언어 이름은 Cypher).

## 4. 폴더 구조 — 어디에 뭐가 있나

웹개발자에게 익숙한 방식으로 매핑하면 이렇습니다.

```
pdf/
├─ backend/                  ← 서버 (FastAPI = Python판 Express/NestJS 느낌)
│  ├─ app/
│  │  ├─ main.py             ← 서버 진입점. 지금은 /health 체크만 있음 (라우터는 아직)
│  │  ├─ core/config.py      ← 환경설정(.env에서 API키, DB주소 읽음) = config 모듈
│  │  ├─ loaders/            ← ① 로딩: 파일형식별로 읽어 표준화
│  │  │   ├─ base.py         ←    공용 데이터 형태(LoadedDoc/Element) 정의
│  │  │   ├─ router.py       ←    확장자 보고 알맞은 로더로 분배 (PDF/MD/DOCX...)
│  │  │   ├─ pdf_loader.py   ←    PDF 전용 (PyMuPDF)
│  │  │   ├─ text_loader.py  ←    MD/TXT 전용
│  │  │   └─ unstructured_loader.py ← DOCX/XLSX/PPTX 등
│  │  ├─ chunking/chunker.py ← ② 청킹: 표 보존하며 적당히 자르기
│  │  └─ graph/              ← ③④ 그래프
│  │      ├─ extractor.py    ←    AI로 엔티티·관계 추출 (Claude 호출)
│  │      └─ store.py        ←    Neo4j 적재 + 인덱스 빌드
│  └─ .env                   ← 비밀키·접속정보 (절대 깃에 올리면 안 됨)
│
├─ scripts/                  ← 손으로 돌려보는 실험 스크립트(PoC)
│  ├─ poc1_loaders.py        ← ① 로딩만 테스트
│  ├─ poc2_chunking.py       ← ②까지 테스트
│  └─ poc3_graph.py          ← ①~④ 전체를 한 번에 돌려 그래프 만들기
│
├─ samples/                  ← 테스트용 가짜 설비 문서 3종 (md/pdf/docx)
├─ eval/dataset.jsonl        ← 채점용 문제지 (질문+정답+출처 10문항)
├─ docker-compose.yml        ← DB 3종을 한 번에 띄우는 설정
└─ docs/                     ← 기획서·작업로그·이 문서
```

> **PoC(Proof of Concept, 개념 증명)** 란? "이 방식이 실제로 되는지 작게 만들어 확인해보는 실험"입니다.
> 정식 기능으로 굳히기 전에, scripts/ 안에서 부분부분 돌려보며 검증하는 단계입니다.

## 5. 인프라 — 왜 데이터베이스가 3개나?

`docker-compose.yml`로 띄우는 3개의 DB는 각자 역할이 다릅니다. (Docker = 프로그램을 통째로 담아 어디서나 똑같이 띄우는 컨테이너 기술)

| DB | 종류 | 이 프로젝트에서의 역할 | 비유 |
|---|---|---|---|
| **Neo4j** | 그래프 DB | 엔티티·관계 지도(지식 그래프) 저장 | 등장인물 관계도 |
| **Qdrant** | 벡터 DB | "의미가 비슷한 문장 찾기"용 (다음 단계에서 사용) | 의미 기반 검색 엔진 |
| **PostgreSQL** | 관계형 DB(SQL) | 문서 메타데이터·사용자·세션 등 일반 데이터 | 익숙한 그 SQL DB |

- **지금(PoC3)은 Neo4j만** 실제로 씁니다. Qdrant/Postgres는 띄워만 두고 다음 단계에서 붙입니다.
- **벡터 검색이 뭐냐**: 글을 AI가 숫자 배열(=벡터)로 바꾸면, 의미가 비슷한 글끼리 숫자도 비슷해집니다. 그 숫자 거리로 "비슷한 내용 찾기"를 하는 게 벡터 검색이고, 그 숫자를 저장·검색하는 전용 DB가 Qdrant입니다. (이번 단계에선 비용·키 문제로 일부러 꺼뒀습니다.)

## 6. 지금 어디까지 됐나 (현재 상태)

전체 12주 계획 중 **1주차(설계 + PoC)** 를 마무리한 상태입니다.

### ✅ 작동하는 것
- **① 로딩**: MD/PDF/DOCX 3종 읽어서 글·표·위치까지 정상 추출 (PoC1)
- **② 청킹**: 표를 안 깨고 적당히 자르기 (PoC2)
- **③④ 그래프**: 매뉴얼 1건을 끝까지 통과시켜 **Neo4j에 지식 그래프 생성 성공** (PoC3)
  - 결과: **노드 24개, 관계 52개**. 펌프/모터/베어링 등이 구동·지지·점검 관계로 연결됨.
- **채점 문제지**: 멀티홉 5 + 단순 5 = 10문항 (`eval/dataset.jsonl`)

### 🚧 아직 안 만든 것
- **벡터 검색 + 하이브리드 검색** (Qdrant 연동) — 다음 단계(W2)
- **실제 질문→답변 엔진** (지금은 그래프를 "만드는" 데까지만. "물어보는" 기능은 없음)
- **API 라우터** — `main.py`에 `/health`(서버 살아있나 확인)밖에 없음. 업로드·채팅 API는 W5+
- **프론트엔드 UI** — 답변+원본 나란히 보여주는 화면, 출처 하이라이트 (W9~10)
- **자동 성능 평가(RAGAS)** — 채점 문제지로 점수 매기는 자동화 (W11~12)

### 한눈에 보는 진행도
```
설계/PoC  ███████████ 100%  ← 지금 여기 (1주차 완료)
파이프라인 ░░░░░░░░░░░   0%   (벡터·하이브리드 검색)
RAG엔진   ░░░░░░░░░░░   0%   (질문→답변)
프론트엔드 ░░░░░░░░░░░   0%   (화면·하이라이트)
평가/배포  ░░░░░░░░░░░   0%
```

## 7. 직접 눈으로 확인해보기

코드를 몰라도 결과는 볼 수 있습니다.

1. Docker Desktop이 켜져 있고 DB 3개가 떠 있는 상태에서,
2. 웹브라우저로 **Neo4j 그래프 화면**을 엽니다:
   - 주소: `http://localhost:7474`
   - 로그인: 아이디 `neo4j` / 비번 `graphrag123`
3. 아래 명령(Cypher 쿼리)을 입력창에 붙이고 실행:
   ```cypher
   MATCH (n)-[r]->(m) WHERE NOT type(r)='MENTIONS' RETURN n,r,m
   ```
4. 그러면 펌프·모터·베어링이 선으로 연결된 **관계도 그림**이 뜹니다. 이게 PoC3가 만든 지식 그래프입니다.

> 글 조각↔엔티티 연결까지 다 보고 싶으면 `WHERE...` 부분을 빼고 `MATCH (n)-[r]->(m) RETURN n,r,m` 만 실행하세요.

## 8. 용어 사전 (이것만 알면 회의 따라감)

| 용어 | 쉬운 뜻 |
|---|---|
| **RAG** | AI가 답하기 전에 우리 문서를 먼저 검색해서 근거 깔고 답하게 하는 기법 |
| **GraphRAG** | 위 RAG에 "관계 지도(지식 그래프)"를 더해 여러 단계 추론을 가능케 한 것 |
| **할루시네이션(환각)** | AI가 모르는 걸 그럴듯하게 지어내는 현상. RAG의 적 |
| **청크(Chunk)** | 문서를 검색·처리하기 좋게 자른 조각 |
| **임베딩/벡터** | 글의 의미를 숫자 배열로 바꾼 것. 비슷한 의미 = 비슷한 숫자 |
| **엔티티(Entity)** | 그래프의 "점". 펌프·모터 같은 대상 |
| **트리플릿(Triplet)** | (주어)-[관계]->(목적어) 형태의 관계 한 줄 |
| **멀티홉 질문** | 한 문장에 답이 없고, 여러 정보를 연결해야 풀리는 질문 |
| **PoC** | 개념 증명. "이 방식 진짜 되나?" 작게 만들어 확인하는 실험 |
| **LlamaIndex** | RAG 파이프라인을 쉽게 짜게 도와주는 프레임워크(라이브러리) |
| **Cypher** | Neo4j 그래프 DB에 질의하는 언어 (SQL의 그래프 버전) |
| **FastAPI** | Python으로 API 서버 만드는 프레임워크 (Express/NestJS 같은 역할) |
